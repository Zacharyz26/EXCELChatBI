# v2.4 阶段 1 实施记录

> 状态：本地实现完成 · 远端镜像门禁待运行 · 更新日期：2026-07-23

本记录只描述已经进入代码的事实和仍未完成的工作。阶段 1 的最终验收仍以
[`Agent自主化开发规划.md`](../Agent自主化开发规划.md) 为准。

## 本批已实现

- SQLite schema 从 v1 升至 v2：正式版本迁移、不可变 checksum 校验、v1 在线备份及 hash
  登记、未知版本拒绝、无活动 TaskRun 时的受保护 v2→v1 回滚；
- 新增 `task_runs`、`task_contracts`、计划/步骤预留表、`task_events`、`task_snapshots`、
  `tool_invocations`、`evidence`、`claims`、关联表和 `checkpoints`；
- 最小 TaskContract 与 AgentState：目标、非空答复条件、明确图表/报告条件、工具调用预算、
  状态版本和终态原因；
- 工具调用写入幂等键、参数 hash、执行状态、结果 hash 和 Evidence；
- 候选最终正文先缓冲，进入 `verifying` 后执行确定性检查，通过后才写最终消息、发送正文并
  标记 `completed`；
- 图表只接受当前 run 中带非空 ECharts option 的 Artifact；报告还要求当前 run 的 Artifact、
  下载引用以及真实存在的非空生成文件；
- 预算耗尽不再伪装成功，TaskRun 进入 `blocked` 并向用户明确说明未完成；
- 旧 SSE 保留，同时增加带 `run_id`、sequence 和 event_id 的 `goal`、`run.started`、
  `verification.started`、`verification` 生命周期事件；
- 增加 `GET /agent/runs/{run_id}` 与 `GET /agent/runs/{run_id}/events` 只读恢复接口；
- 用户消息、Conversation 更新时间、TaskRun、TaskContract、首个 `goal` 事件与初始快照已在
  同一个 SQLite `BEGIN IMMEDIATE` 事务中创建；中途失败时不再留下孤立用户消息；
- 工具成功 Evidence 增加有界的数值路径索引；Finalizer 前从候选答复提取最小数值 Claim，
  持久化 `value_refs` 与 Claim→Evidence 关联，并校验 Evidence 必须属于当前 run；
- 候选答复包含无 Evidence 支持的数字时不会直接交付：Verifier 返回
  `unsupported_numeric_claim`，循环允许一次受约束纠正，仍不满足则明确失败；
- 数值匹配要求 Evidence 中存在同值表示，不把比率自动换成百分比，也不允许显示舍入冒充
  工具结果；用户目标中明确的时间范围数字不作为新结论处理；
- 新的唯一成功入口 `commit_tool_success` 在一个事务中提交 Artifact、Invocation succeeded、
  Evidence、`step.completed`、TaskSnapshot 和 Checkpoint，并同步递增 `state_version`；旧完成入口
  已收窄为只允许 `failed/unknown`，不能绕过原子成功边界；
- Artifact 与 `step.completed` 只在成功事务提交后发送；单独删除被 Evidence/Invocation 引用的
  Artifact 会被拒绝，删除整个 Conversation/Project 时先清理 TaskRun 证据图再安全级联；
- Markdown/PDF 改为同目录临时文件写入后原子 rename；组合 PDF 失败或 Agent 数据库提交失败时
  清理由该次操作生成且位于配置报告目录内的文件。
- 新增版本化的受约束语义 Verifier 协议：模型只能逐成功标准判断 Claim 覆盖、越界表述与
  局限披露，不能返回最终任务 verdict，也不能覆盖任何确定性失败；最终
  `PASS/NEEDS_ACTION/WAITING_USER` 仍由代码推导；
- 语义响应使用严格 JSON Schema、精确 criterion/Claim/Evidence ID 集合与引用校验；硬检查未
  通过时完全不调用模型，非法结构或模型自相矛盾会作为协议错误失败关闭；
- 模型 registry 增加稳定候选枚举和无 fallback 的隔离评测路由，避免把 fallback 成功错误记到
  被测模型；新增 14 个正反配对语义 fixture 和可重复 CLI，报告只保存请求/响应 hash、解析结果、
  实际模型、token、延迟、成本可用性与错误类型，不持久化原始模型响应；
- 真实 DeepSeek 单轮评测已给出明确 `NO_GO`，因此语义 Verifier **未接入生产结束条件**，当前
  生产路径继续只使用确定性 Verifier。
- 新增 `tool-policy-v1` 中央策略网关：每次工具执行前确定性检查主体、静态工具 allowlist、
  项目资源范围和调用预算；策略上下文由 Host 构造，模型参数不能提供或覆盖身份与权限；
- 策略允许/拒绝都会产生有界结构化审计事件，只记录规则版本、主体、资源、参数 hash 和
  permission snapshot，不记录原始工具参数、数据或密钥；当前审计后端为结构化日志，企业级
  append-only 存储仍属于 v3.0；
- `step.started` 与 running Invocation 已在同一事务提交；工具失败、策略拒绝和未知结果通过
  `commit_tool_failure` 原子提交 Invocation、`step.completed`、正式 Observation、TaskSnapshot
  和 `state_version`，失败不生成 Evidence；
- 工具成功的 `step.completed` 也包含 `Observation(status=ok)`，并以 Evidence ID 作为受控
  payload reference；Invocation=`unknown` 时确定性 Verifier 强制返回 `BLOCKED`，必须先按
  幂等键对账，模型不能提前结束流程；
- 模型轮次和工具执行新增结构化 trace span，记录 trace/span/run/invocation、模型、token、
  延迟、成本可用性、工具与结果类型；不记录原始消息、整表或工具结果。
- `mcp==1.28.0` 已精确锁定；新增传输无关的 Tool Capability Contract、Host-owned
  RequestContext、稳定错误和结构化结果协议，15 个项目底层工具及 Agent 的 11 个模型工具均从
  同一份输入/输出 schema 与治理元数据生成 MCP 描述；
- 官方 MCP Python SDK low-level Server adapter 已实现 `tools/list`/`tools/call`，调用缺少 Host
  上下文、输入非法、输出非法或工具未知时失败关闭；stdio 服务入口已接入现有各工具服务；
- Client Gateway 已实现静态 allowlist、发现集合与契约 hash 校验、官方 SDK Session transport
  及不健康 Server 调用阻断；生产仍执行一次原进程内 runner，影子层只校验 schema、结果 hash、
  错误映射和 Artifact 类型，不重复制造数据集、文件或 Artifact；
- 新增 API Python 多阶段镜像、Web 构建/非特权 Nginx 镜像、`.dockerignore`、持久状态目录、
  liveness 与 `/api` SSE/下载代理；GitHub Actions 已增加镜像构建、非 root 启动和健康 smoke job。
  当前开发机没有 Docker daemon，因此首次真实镜像构建仍需远端 CI 结果关闭门禁。
- Evidence 摘要增加有界 `source_index` 和知识检索空结果标记；生产 Finalizer 现同时提取数值、
  知识与显式局限 Claim。知识回答必须引用本次 `kb_search` 实际返回的 source；检索无命中时只
  接受明确的“未找到/无法回答”，不能用肯定式业务定义结束任务；Claim 记录显式来源匹配或
  限定表述的置信类型。
- 同一数值出现在多个 Evidence 路径时不再静默把第一条当成唯一语义来源：优先选择与结论字段
  语义匹配的路径，并把全部候选路径写入 `candidate_paths` 供审计。
- 新增报告 Artifact/文件系统对账：API 启动时只读检查缺失引用、未登记发布文件、硬崩溃遗留
  临时文件和越界 `file_ref`；运维脚本默认只审计，只有显式参数才会删除超过宽限期且严格匹配
  原子写入命名的临时文件，绝不自动删除可能来自旧兼容端点的发布报告。
- 图表/报告意图正则与通用 Artifact 后置条件的固定影子集已覆盖中英文、否定、PDF、组合请求
  等 9 个场景，达到 9/9 等价；正则现定位为阶段 1 的保守意图编译兼容层，最终是否完成始终只
  由通用 TaskContract/Verifier 后置条件决定。

## 本批验证

- schema 新建、v1→v2 备份迁移、checksum 篡改拒绝、回滚活动任务保护；
- TaskRun 状态版本冲突、事件序列、调用幂等和成功结果 Evidence；
- 任务启动事务成功与强制失败回滚、跨 run Evidence 引用拒绝、Claim 覆盖更新；
- 多数值路径绑定、无依据数字、目标时间范围、目标同值结果、百分比换算拒绝；
- “模型停止但没有图表”、PDF URL 存在但文件不存在、工具成功无 Evidence、预算耗尽等
  Verifier 反例；
- 原 Agent 循环的工具、图表、报告、失败重试、熔断、预算和上下文用例，以及“模型先报
  错数字、Verifier 纠正后再交付”的端到端回归；
- Artifact 已插入后强制 Evidence 写入失败的故障注入，确认 Artifact、Invocation、Evidence、
  Event、Snapshot 和 Checkpoint 全部保持提交前状态；
- Evidence 引用删除保护、整段 Conversation 级联删除、报告原子发布及提交失败文件清理。
- 受约束语义 Verifier 的硬失败短路、严格响应协议、未知引用拒绝、代码派生 verdict，以及
  评测集重复 ID/配对覆盖/false PASS 计分和模型隔离路由；相关定向套件 33 个测试通过；
- `semantic-verifier-v1` 的 V3 烟测发现输出协议歧义并被严格拒绝；收紧为 v2 后，V3 与 R1
  均通过 SV01/SV02 正反烟测；随后各自运行 14 场景一轮完整评测：V3 为 11/14、3 次 false
  PASS，R1 为 11/14、2 次 false PASS、1 次 false block，均无协议/模型错误，结论为
  `NO_GO`。V3 平均约 1.62 秒/123 completion tokens，R1 平均约 6.91 秒/752 completion
  tokens；成本因网关尚未配置单价而记为 unavailable。
- 策略允许/未准入工具/跨项目资源/预算耗尽/非法上下文，审计字段最小化，模型与工具 trace，
  生产 Agent 的执行前拒绝，以及 `step.started`、失败/未知 Observation 的状态版本与事件顺序；
- 对失败 Observation 的事件插入做故障注入，确认 Invocation、TaskRun 版本和 Snapshot 全部回滚；
  unknown Invocation 即使模型声称完成也会被确定性 Verifier 阻塞；
- 官方 SDK 内存会话完成 initialize、`tools/list` 和带 Host metadata 的 `tools/call` 往返；验证
  缺上下文、输入 schema、输出 schema、未知工具、发现契约漂移及 Artifact 后置条件不一致；
- 全部 15 个底层项目工具均带 capability、版本、风险、权限、幂等和后置条件元数据；API/Web
  容器静态契约、前端 TypeScript 检查和生产构建通过；真实 Docker build 由新增远端 CI job 验证；
- 第六个切片结束时全量后端回归在固定 `hashing/lexical/local` RAG 测试后端下为
  274 passed、2 skipped；
  Ruff 全库通过，本批 Python 源文件定向 mypy 通过。全库 mypy 仍有 43 个既有类型债务，未在
  本批扩大范围处理。
- 本次收尾新增知识 source、空检索诚实回答、同值多路径、局限 Claim、报告目录对账与正则影子
  门禁测试；阶段 1 定向回归 58 passed；最终全量后端回归更新为 290 passed、2 skipped，Ruff
  通过，7 个本次改动源文件 mypy 通过，前端 TypeScript 检查和生产构建通过。
- 对当前真实报告目录执行只读对账：8 个已登记文件引用均存在、无越界引用、无遗留临时文件；

## 阶段 1 收尾结论与未关闭门禁

1. 阶段 1 的本地代码、静态契约和回归门禁已经完成；当前唯一直接属于阶段 1 验收且尚未取得
   运行证据的项目，是 API/Web 镜像首次真实构建、非 root 启动和健康 smoke。开发机没有 Docker
   daemon，且本批 CI/Docker 改动尚未提交到远端，因此必须在提交推送后的 GitHub Actions 中关闭，
   在此之前不能把阶段 1 标记为“正式验收通过”。
2. 受约束语义 Verifier 的实现、失败关闭协议和 go/no-go 决策已完成；首轮真实评测因 false PASS
   得出 `NO_GO`，所以生产保持禁用是安全门禁的正确结果，不是用不可靠模型凑齐功能。下一候选、
   新 heldout、三次重复与人工盲评属于阶段 0 的后续能力验证债务。
3. Planner/v2.3 行为基线、MCP stdio+Streamable HTTP 双传输探针及设计评审仍是阶段 0 未关闭项，
   不因阶段 1 本地实现收尾而自动视为通过。
4. TaskPlan/TaskStep、动态重规划、澄清、暂停/恢复/取消、MCP 规范执行切换、工具镜像、根 Compose、
   卷恢复和容器 E2E 均按现行路线图属于阶段 2，不列作阶段 1 实现缺陷。
5. 真实用户/租户鉴权、审批和企业级 append-only 审计属于后续治理范围；阶段 1 已完成本地主体、
   静态 allowlist、中央策略、结构化审计和 trace 骨架。
