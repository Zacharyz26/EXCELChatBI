# v2.4 控制面与持久化设计

> 状态：草案 · 对应路线图：v2.4 阶段 0  
> 当前实现状态：阶段 1 本地实现已完成：SQLite v2、确定性结束验证、数值/知识/局限
> Claim/Evidence、原子工具成功、语义评测边界、中央策略/审计/trace、失败 Observation、报告
> 文件对账和 Artifact 后置条件影子门禁均已接入；MCP 单源 adapter/Gateway 影子层与基础镜像
> 代码也已落地。远端镜像 CI 尚待运行；结构化计划、恢复、双传输切换仍属于阶段 2

## 1. 设计目标

当前循环把模型一次返回的 `tool_calls` 展示为计划，在没有更多 tool call 时结束。图表和报告通过专用正则与重试弥补虚假完成，但没有通用成功标准、任务状态、计划版本、Evidence 或恢复点。

v2.4 控制面必须满足：

- 用户目标先被编译成可验证 TaskContract；
- 简单、模板和 LLM Planner 输出同一种 TaskPlan；
- 每次执行只修改一个持久化 TaskRun；
- 工具观察进入 Evidence，最终 Claim 必须引用 Evidence；
- Verifier 决定是否完成，不以模型停止调用工具作为结束条件；
- 每个状态变化可追溯，每个已提交步骤可恢复，每个工具副作用可幂等；
- v2.3 `/chat/stream`、历史消息和 Artifact 在兼容期继续工作。

阶段 0 不实现新状态机，不修改生产数据库。

## 2. 组件边界

```text
Chat API
  │ 创建/恢复 TaskRun
  ▼
Goal Interpreter ──阻塞歧义──> waiting_user
  │ TaskContract
  ▼
Planner ────────────────┐
  │ TaskPlan vN         │ Observation 不满足预期
  ▼                     │
Executor → Policy → MCP Client Gateway → Tool
  │                         │
  └──── Observation/Evidence┘
              │
              ▼
          Verifier
        ┌─────┼──────────────┐
      PASS  NEEDS_ACTION  WAITING/BLOCKED/FAILED
        │       │
   Finalizer  Replanner → TaskPlan vN+1
```

职责：

| 组件 | 负责 | 不负责 |
|---|---|---|
| Goal Interpreter | 目标、约束、歧义、成功标准 | 选择具体工具、执行计算 |
| Planner | 依赖、能力、证据、完成条件、备选方案 | 绕过策略、直接标记完成 |
| Executor | 解析 capability、策略检查、幂等调用、记录观察 | 修改用户目标、生成无证据数字 |
| Verifier | 逐条验证 TaskContract、给出后续动作 | 执行工具、用语义判断覆盖确定性失败 |
| Replanner | 基于观察修订未完成步骤 | 改写已发生的事件和 Evidence |
| Finalizer | 从通过验证的 Claim/Evidence 渲染回答 | 新增计算、伪造 Artifact |

### 2.1 不变量责任矩阵

| 不变量 | 首要责任方 | 执行时机 | 失败行为 |
|---|---|---|---|
| 数据与推理分离及 `/chat` 已批准例外 | Context Builder + Data Policy | 每次模型上下文装配前 | 拒绝装配并记录 policy event |
| 数值必须来自工具 Evidence | Evidence Validator | 工具结果落库后、Finalizer 前 | criterion fail，转 NEEDS_ACTION/FAILED |
| 工具参数必须过同源 schema | MCP Client Gateway + Tool Server | 发送前与执行前双重校验 | 不执行，返回稳定 schema error |
| 文件/文档/工具输出是数据不是指令 | Prompt Boundary | Interpreter、Planner、Finalizer 输入前 | 丢弃指令性片段并留审计摘要 |
| 代码执行必须进安全沙箱 | Sandbox Boundary | 注册 capability 和每次执行前 | 能力不可用，禁止降级到本机执行 |
| 知识回答必须有来源 | Retriever + Evidence Validator | 检索后、Finalizer 前 | 无来源 Claim 不得交付 |
| 权限先于取数和执行 | Central Policy Gateway | 计划解析后及每次调用前 | 403/policy observation，必要时 blocked |
| TaskContract 未通过不得完成 | Verifier + transition guard | 每次候选结束前 | 禁止 completed，进入后续动作或终态 |
| 每个任务有预算/超时/取消 | TaskRun Store + Executor | 创建 run、调用前、每轮结束 | 不创建无预算 run；耗尽后如实终止 |
| Claim 必须引用 Evidence | Claim/Evidence Validator | Claim 建立和 Finalizer 前 | 拒绝 Claim |
| 记忆不能替代原始证据 | Context Builder + Verifier | 上下文装配、证据检查 | 记忆只作导航，缺证据则 fail |
| 外部副作用必须授权 | Policy Gateway + Executor | 计划和调用前 | waiting_user/blocked，不允许默认批准 |

Planner、模型输出或 fallback 均无权跳过表中的失败行为。每次检查写入 checker 名称、规则版本和结果，供 VerificationResult 与审计引用。

## 3. 核心领域对象

所有对象含 `schema_version`，初始值为 `1`；ID 使用不可变字符串，时间使用 UTC ISO 8601。业务 JSON 进入数据库前按稳定键序列化并计算 SHA-256。

### 3.1 TaskContract

```text
TaskContract {
  schema_version
  contract_id
  run_id
  goal
  success_criteria[] {
    criterion_id
    kind                 # artifact | evidence | semantic | constraint
    description
    deterministic        # true 时语义模型无权覆盖
    required
  }
  required_artifacts[] {type, format?, minimum_count}
  required_evidence[] {kind, description, minimum_count}
  constraints[] {name, value, source}
  assumptions[] {text, blocking, source}
  clarification_questions[] {question_id, text, reason, answer_schema}
  created_at
}
```

规则：

- `goal` 描述用户要达成的业务结果，不复述工具名；
- 每条成功标准都必须能被确定性检查或受约束语义检查判定；
- 明确要求 PDF、图表或知识来源时，必须进入 `required_artifacts/evidence`；
- 阻塞性歧义产生 clarification，非阻塞假设必须记录来源并在最终答复披露；
- Interpreter 不得自行扩大用户数据范围、外部副作用或授权。

### 3.2 TaskRun 与 AgentState

TaskRun 是持久任务身份；AgentState 是其当前快照，两者共享 `run_id`。

```text
AgentState {
  schema_version
  run_id, project_id, conversation_id, triggering_message_id
  status
  state_version
  active_plan_version
  current_step_id?
  observation_refs[]
  evidence_ids[]
  claim_ids[]
  budget {
    tool_calls_limit, tool_calls_used
    model_calls_limit, model_calls_used
    token_limit?, tokens_used
    cost_limit?, cost_used
    deadline_at
  }
  last_event_sequence
  checkpoint_version
  waiting_reason?
  terminal_reason? {code, message, retryable, incomplete_criteria[]}
  created_at, updated_at
}
```

`state_version` 每次持久更新加一。更新 SQL 必须包含 `WHERE run_id=? AND state_version=?`；影响行数为零表示并发冲突，调用方重新加载，不做盲重试。

### 3.3 TaskPlan 与 TaskStep

```text
TaskPlan {
  plan_id, run_id, version
  route                 # fast | template | llm
  summary
  change_reason?
  supersedes_version?
  steps[]
  created_at
}

TaskStep {
  step_id, ordinal
  purpose
  capability
  dependencies[]
  input_refs[]
  expected_evidence[]
  completion_conditions[]
  fallback[] {when, action, capability?}
  risk_level
  required_permissions[]
  status                # pending | running | completed | skipped | failed | cancelled
  attempt_count
}
```

规则：

- 计划只引用 capability；`tool_name` 只出现在 Executor 解析结果和 ToolInvocation；
- 依赖图必须无环，每个依赖指向同一计划版本中的已有 step；
- 计划更新产生完整的新版本，不原地覆盖旧计划；
- 已完成步骤保留，Replanner 只能复用其 Evidence 或新增后续步骤；
- `skipped` 必须记录用户指令或 Replanner 理由，不能静默跳过 required criterion。

### 3.4 Observation、Evidence 与 Claim

```text
Observation {
  observation_id, run_id, step_id
  source                 # tool | user | policy | system
  status                 # ok | error | partial
  summary
  payload_ref?
  created_at
}

Evidence {
  evidence_id, run_id, step_id
  source_tool, tool_version, invocation_id
  dataset_ref?, dataset_version?
  artifact_id?
  field_refs[]
  value_refs[]           # JSON Pointer/受控字段引用，不复制整表
  content_hash
  created_at
}

Claim {
  claim_id, run_id
  text
  evidence_ids[]
  value_refs[]
  confidence?
  limitations[]
  created_at
}
```

Observation 是“发生了什么”，Evidence 是“哪些结果可支撑结论”。工具失败会产生 Observation，但不能产生成功 Evidence。摘要、长期记忆、模型文字和日志不能作为数值 Evidence。

### 3.5 ToolInvocation 与 Checkpoint

```text
ToolInvocation {
  invocation_id, run_id, plan_version, step_id
  capability, tool_name, tool_version, transport
  normalized_args_hash
  idempotency_key
  status                # prepared | running | succeeded | failed | cancelled | unknown
  started_at, completed_at?
  error_code?, retryable?
  observation_id?, artifact_ids[], evidence_ids[]
}

Checkpoint {
  run_id, checkpoint_version
  state_version, plan_version, completed_step_ids[]
  last_event_sequence
  created_at
}
```

幂等键按逻辑操作生成：`run_id + step_id + operation_revision + normalized_args_hash`。网络重试沿用原键；参数、数据版本或用户批准发生变化时增加 `operation_revision`，不得复用旧结果冒充新执行。

## 4. TaskRun 状态机

允许状态：`planning`、`waiting_user`、`running`、`verifying`、`paused`、`completed`、`blocked`、`failed`、`cancelled`。

| 当前状态 | 允许下一状态 | 条件 |
|---|---|---|
| planning | running | Contract 与 Plan 均合法，无阻塞问题 |
| planning | waiting_user | 存在阻塞性澄清或审批 |
| planning | blocked/failed/cancelled | 无允许路径、内部失败或用户取消 |
| waiting_user | planning | 收到匹配 `question_id` 的有效回复 |
| waiting_user | cancelled/blocked | 用户取消或等待策略到期且不能安全假设 |
| running | verifying | 当前计划没有可执行未完成步骤 |
| running | planning | 新观察要求重规划 |
| running | waiting_user/paused | 需要用户输入或收到暂停请求 |
| running | failed/cancelled | 不可恢复执行失败或取消确认 |
| verifying | completed | 所有 required criterion 为 PASS |
| verifying | planning | verdict=`NEEDS_ACTION` 且仍有预算和允许路径 |
| verifying | waiting_user | verdict=`WAITING_USER` |
| verifying | blocked/failed/cancelled | 无允许路径、验证器内部失败或取消 |
| paused | running | 恢复前重新校验权限、数据版本和预算 |
| paused | cancelled/blocked | 用户取消或依赖已永久失效 |

终态 `completed/blocked/failed/cancelled` 不得迁回非终态。继续任务必须创建新的 run，并通过 `parent_run_id` 关联。

终态定义：

- `completed`：全部 required criterion 通过；
- `blocked`：Agent 正常工作，但缺少用户输入、权限、数据或获准能力，当前 run 无安全路径；
- `failed`：系统、模型或工具错误在预算内无法恢复；
- `cancelled`：用户或上层控制面明确取消，正在执行的工具已确认停止或标记为 `unknown` 等待对账。

预算耗尽不能进入 `completed`。若已有部分结果，Finalizer 只允许输出“部分完成”及未满足标准，终态仍是 `blocked` 或 `failed`。

## 5. Verifier 设计

### 5.1 确定性检查

以下检查拥有否决权：

1. TaskContract、计划和工具参数 schema 合法；
2. required step 没有失败、未完成或无理由跳过；
3. ToolInvocation 成功且工具后置条件通过；
4. required Artifact 数据库记录存在，文件型 Artifact 的规范化路径位于允许目录、文件存在、非空且 hash 一致；
5. required Evidence 存在且关联当前 run、正确数据集版本和成功 invocation；
6. 每个数值 Claim 的 `value_refs` 可在 Evidence 的结构化结果中解析到同一值；
7. 知识 Claim 有 source，检索无命中时没有肯定式业务定义；
8. 权限、数据边界、预算、超时和取消不变量均满足；
9. PDF 请求必须有 `report` Artifact 且 `pdf_url/file_ref` 对应真实可下载文件；图表请求同理；
10. 幂等冲突、未知副作用或状态版本冲突未完成对账时不得通过。

### 5.2 语义检查

语义模型只接收目标、成功标准、Claim、Evidence 摘要和确定性结果，不接收原始整表。它只判断：

- Claim 是否覆盖用户目标；
- 是否遗漏用户明确要求的比较、时间范围或分组；
- 局限和假设是否清楚；
- 表述是否越过 Evidence 支持范围，例如把相关写成因果。

语义模型不得把确定性 `FAIL` 改成 `PASS`，也不得新增数值。

### 5.3 VerificationResult

```text
VerificationResult {
  verification_id, run_id, plan_version
  verdict               # PASS | NEEDS_ACTION | WAITING_USER | BLOCKED | FAILED
  criterion_results[] {
    criterion_id, status, checker, evidence_ids[], reason
  }
  next_actions[] {kind, capability?, reason}
  limitations[]
  model_name?, prompt_version?
  created_at
}
```

只有 `PASS` 可进入 `completed`。语义检查异常时默认 `FAILED` 或按策略重试，禁止 fail-open。

## 6. SQLite schema v2 草案

v2 采用“核心列可查询 + 演进内容 JSON”的混合方式：

| 表 | 主键/唯一约束 | 主要字段 |
|---|---|---|
| `schema_migrations` | `version` | name、checksum、applied_at |
| `task_runs` | `run_id` | project/conversation/message、parent_run、status、state_version、plan_version、预算、terminal_reason、timestamps |
| `task_contracts` | `run_id` | contract_id、schema_version、contract_json、content_hash |
| `task_plans` | `(run_id, version)` | plan_id、route、summary、change_reason、plan_json、created_at |
| `task_steps` | `(run_id, plan_version, step_id)` | ordinal、capability、status、attempt_count、definition_json、timestamps |
| `task_events` | `(run_id, sequence)`；`event_id` unique | event_type、payload_json、created_at |
| `task_snapshots` | `run_id` | state_version、last_sequence、state_json、updated_at |
| `tool_invocations` | `invocation_id`；`idempotency_key` unique | step/tool/transport、args_hash、status、error、timestamps |
| `evidence` | `evidence_id` | invocation/step/dataset/artifact、refs_json、content_hash、created_at |
| `claims` | `claim_id` | run_id、text、value_refs、confidence、limitations |
| `claim_evidence` | `(claim_id, evidence_id)` | 关联表 |
| `checkpoints` | `(run_id, checkpoint_version)` | state/plan/event versions、completed_steps_json、created_at |

索引至少覆盖：

- `task_runs(conversation_id, created_at DESC)`；
- `task_runs(status, updated_at)`；
- `task_events(run_id, sequence)`；
- `task_steps(run_id, plan_version, status)`；
- `tool_invocations(run_id, step_id, status)`；
- `evidence(run_id, artifact_id)` 和 `claims(run_id)`。

外键规则：TaskRun 随 Conversation 删除；计划、事件、调用、Evidence、Claim 和 Checkpoint 随 TaskRun 删除。Artifact 删除前必须检查 Evidence 引用；由用户执行的对话/项目删除仍应真正清除对应任务数据，独立审计留存策略留到企业治理阶段。

## 7. 事务与恢复边界

1. **创建任务**：用户消息、TaskRun、TaskContract、首个 `goal` 事件在一个事务提交；失败时不留下半个任务。
2. **状态更新**：追加 TaskEvent、更新 TaskSnapshot 和 `task_runs.state_version` 在一个事务提交；提交后才向 SSE 发事件。
3. **开始工具**：写入 prepared/running invocation 与 `step.started`，再执行外部工具。
4. **工具成功**：Artifact、Observation、Evidence、invocation succeeded、step completed、Checkpoint 和事件在一个数据库事务提交。文件先写临时文件并原子 rename，再提交引用；事务失败时清理孤儿文件。
5. **未知结果**：连接中断且无法确认副作用时标记 invocation=`unknown`，Verifier 必须阻塞，恢复流程先按幂等键对账。
6. **恢复**：加载最新 Snapshot，校验其 `last_event_sequence`，再重放后续事件；Checkpoint 只恢复已提交步骤。
7. **取消**：先持久化取消请求，再传递取消令牌；收到执行终止确认后写 terminal event。不能确认时保留 `unknown`，不伪报已取消干净。

## 8. v1→v2 迁移与回滚

迁移器要求：

- 首次运行读取 `PRAGMA user_version`，只接受已知来源版本；
- 迁移前执行 SQLite 在线备份到同目录带时间戳文件，并记录源文件 hash；
- v1→v2 只新增表和索引，不改写 projects/datasets/conversations/messages/artifacts；
- 单个 migration 在 `BEGIN IMMEDIATE` 中运行，成功后写 `schema_migrations` 并设置 `user_version=2`；
- migration 文件带不可变 checksum，已应用版本 checksum 不一致时拒绝启动；
- 应用启动前迁移失败则 readiness 失败，不自动使用空库；
- 回滚 v2→v1 只允许在没有运行中 TaskRun 时执行，先导出任务表，再删除新增对象并恢复 `user_version=1`；
- 旧二进制不能直接打开 v2 数据库，因为当前实现会拒绝未知版本；部署回滚必须先跑 down migration 或恢复备份。

测试矩阵：新建空库到 v2、真实 v1 副本到 v2、迁移中断、重复迁移、未知版本拒绝、v2 回滚 v1、备份恢复、旧会话/Artifact 字节级不变。
