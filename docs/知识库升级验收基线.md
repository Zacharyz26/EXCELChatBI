# 知识库升级验收基线（bge-m3 / Milvus Lite 并行轨）

> 依据：设计文档 13.6 步骤 0/1、13.2 决策 1（bge-m3 稠密+稀疏）/ 决策 2（Milvus Lite
> 起步，换 standalone 只改 URI）/ 决策 4（device 配置项）。
> 本文是该并行轨的**唯一验收依据**；替身链路（hashing/lexical/local）长期保留为
> 默认后端与回归基线，升级不删替身。

## 1. 范围

- 填实 `BGEEmbedder`（bge-m3 稠密+稀疏单次编码）与 `BGEReranker` 两个存根。
- 新增 `MilvusKnowledgeStore`（Milvus Lite，本地文件 URI；含稀疏向量字段）。
- 检索层稀疏路：embedder+store 都支持时走 bge-m3 lexical weights，否则回退中文 BM25。
- 配置项：`RAG_EMBEDDER/RAG_RERANKER/RAG_STORE/MILVUS_URI/EMBEDDING_DEVICE/RAG_MIN_RELEVANCE`。
- 不做：检索之外的任何 Agent/端点改动；语料管理后台；增量索引。

## 2. 环境准备与已验证事项（步骤 0 spike 结论）

| 事项 | 结论 |
|------|------|
| Milvus Lite 稀疏向量 | **已实机验证可用**（pymilvus 2.6 + milvus-lite 3.1：SPARSE_FLOAT_VECTOR + SPARSE_INVERTED_INDEX 的写入/检索/删除全通过，2026-07-17） |
| pymilvus 依赖名 | 2.6 起必须 `pymilvus[milvus_lite]`（已写入 pyproject `--extra rag`），否则本地 URI 连接报错 |
| 代理环境（WSL 常见） | Milvus Lite 走本机 gRPC，系统代理会劫持 127.0.0.1：需 `NO_PROXY=127.0.0.1,localhost`（测试内已内置） |
| pymilvus 主键返回键名 | 2.6 起 search 结果主键在**主键字段名**下（非 `id`）；解析为空会让 RRF 按空串折叠成单候选，检索退化为稠密 top1（已修并有回归测试） |
| FlagEmbedding + torch | 已实测可用（torch 2.12 cu130，RTX 5070 走 cuda+fp16）；**transformers 必须 <5**（5.x 移除 `prepare_for_model`，FlagReranker 崩，已钉入 pyproject） |
| 权重离线侧载 | `EMBEDDING_MODEL`/`RERANK_MODEL` 支持本地目录路径；有网机器预下载 BAAI/bge-m3 与 BAAI/bge-reranker-v2-m3（各约 2.3GB）后拷入，配 `HF_HUB_OFFLINE=1` |
| device 切换 | `EMBEDDING_DEVICE=auto|cpu|cuda`，auto 按 torch 探测；cuda 下自动 fp16 |

## 4. 指标与验收阈值

执行器：`uv run python scripts/kb_eval.py`（`--enforce` 按阈值判定退出码）。
评测索引写临时目录，不污染真实索引。

| 指标 | 替身基线（2026-07-17 实测） | bge 验收线 | bge 实测（2026-07-20，RTX 5070 cuda+fp16） |
|------|------|------|------|
| lexical hit@3 | 100%（hit@1 86%） | **= 100%**（不得低于替身） | **100%**（hit@1 86%）✅ |
| semantic hit@1 | 70% | **≥ 90%** | **90%** ✅ |
| semantic hit@3 | 90% | **= 100%** | **100%** ✅ |
| 负例拒答率 | 100% | **= 100%**（阈值标定后复测） | **100%** ✅ |
| 单查询端到端延迟 | avg 1ms | CPU ≤ 2.5s / GPU ≤ 0.5s | **avg 143ms · max 1.2s（含首跑预热）** ✅ |
| 摄入吞吐（16 片段） | 0.34s | 记录实测值即可，无硬线 | 2.0s |

**已标定阈值：`RAG_MIN_RELEVANCE=0.02`**（2026-07-20；正例 top1 min 0.036 / 负例 top1
max 0.005，取分离带中点）。`--enforce` 全部达标，验收通过。

**工程验收（与指标同为必过项）：**

1. 替身回归绿：`RAG_EMBEDDER=hashing RAG_RERANKER=lexical RAG_STORE=local` 下
   `uv run pytest` 全绿（升级不破坏默认链路）。
2. device 切换不改代码：仅改 `EMBEDDING_DEVICE` 在 cpu/cuda 各跑一次 eval。
3. 向量库切换只改 URI：`MILVUS_URI` 指向 standalone 地址后 eval 可跑（有条件时验证）。
4. 离线可用：断网 + `HF_HUB_OFFLINE=1` + 本地权重目录可完成摄入与检索。
5. `/kb/ingest`、`/kb/query`、Agent `kb_search` 在 bge+milvus 配置下手工冒烟通过；
   切换存储后端后需重摄入（`POST /kb/ingest`，幂等）。

## 5. RAG_MIN_RELEVANCE 阈值标定方法

**阈值语义（2026-07-20 修正）**：阈值按 **top1 重排分**判定"知识库是否有相关内容"
（红线6 的拒答门槛），**不做逐条过滤**——一旦 top1 过线，重排序列即最终排序。
次位相关命中的绝对分天然偏低，逐条截断会误伤（真实回归过：排第 3 的相关片段被
0.02 阈值切掉导致 hit@3 失分）。该语义对任何语料成立，不依赖个例调参。

reranker 换成 bge 后分数分布完全不同（sigmoid 归一到 0-1），沿用替身阈值会失真：

1. 跑 `uv run python scripts/kb_eval.py`，读输出末尾的分数分布：
   - `正例 top1 重排分 min / median`
   - `负例 top1 重排分 max`（漏拒答的负例）
2. 取 **正例最低分与负例最高分的中点** 作为 `RAG_MIN_RELEVANCE` 初值
   （bge normalize 分数经验起点约 0.3，以实测分布为准）；
3. 写入 `.env` 后复跑 eval，确认负例拒答率回到 100% 且正例命中不受损；
4. 阈值与标定日期记录到本文件第 4 节表格下方（保持可追溯）。

## 6. 验收流程

```bash
# 1) 安装重依赖（一次性，约 2GB）
uv sync --extra rag

# 2) （离线环境）侧载权重并配置
#    EMBEDDING_MODEL=/path/to/bge-m3  RERANK_MODEL=/path/to/bge-reranker-v2-m3  HF_HUB_OFFLINE=1

# 3) 切换配置
#    RAG_EMBEDDER=bge  RAG_RERANKER=bge  RAG_STORE=milvus  EMBEDDING_DEVICE=auto

# 4) 跑评测 → 按第 5 节标定 RAG_MIN_RELEVANCE → 复跑带判定
uv run python scripts/kb_eval.py
uv run python scripts/kb_eval.py --enforce   # 全部达标退出码 0

# 5) 替身回归 + 全量测试
uv run pytest

# 6) 服务冒烟：重摄入 + 提问带引用
curl -X POST localhost:8000/kb/ingest -H 'Content-Type: application/json' -d '{"path":"docs/kb_samples"}'
curl -X POST localhost:8000/kb/query  -H 'Content-Type: application/json' -d '{"question":"用户第二天还回来使用的比例叫什么指标"}'
```

全部通过后：把 bge 实测数与标定阈值回填第 4/5 节，提交并在 README 更新
"知识库问答"条目（替身 → bge-m3 语义检索）。
