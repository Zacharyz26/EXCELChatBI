## 1. 背景与目标

构建面向业务数据分析的对话式智能体（ChatBI），用户通过**中文自然语言**完成知识查询、表格数据分析与可视化报告生成，并能对结果持续追问。

### 1.1 核心功能需求

| 编号 | 需求 | 说明 |
|------|------|------|
| F1 | 知识库问答 | 基于企业知识库（业务指标定义、口径文档）的中文检索增强问答 |
| F2 | Excel 自动分析 | 上传表格后自动完成数据分析，产出可视化图表报告 |
| F3 | 多轮 + 多模态 + 追问 | 上下文连续对话、图像理解，并对已生成图表报告追问 |
| F4 | 高级统计分析 | 趋势分析、异常检测、回归分析等 |
| F5 | MCP 工具调用 | 通过 MCP 协议调用内部数据获取工具 |

### 1.2 关键设计原则

1. **数据与推理分离**：LLM 负责理解意图、选择方法、解读结果；所有数值计算交由工具执行，杜绝幻觉数字。
2. **能力工具化**：分析能力以标准 MCP 工具暴露，可独立部署、扩缩容、复用。
3. **编排分层**：标准流程用简单直调编排，复杂流程用状态机/循环编排保证控制力（v2.1：全部自研代码，低代码路线已放弃，见 5.2）。
4. **模型可插拔**：模型层与业务逻辑解耦，按场景路由到不同模型，便于替换与成本控制。
5. **中文优先**：所有涉及语义理解的组件（embedding/rerank/分词/解析）按中文场景选型。
6. **安全前置**：能力暴露的同时，权限与校验作为独立环节强制经过。

## 2. 总体架构

系统采用五层架构。核心特点：**自研编排层**（v2.1 起放弃 Dify）、**可插拔模型路由**、**独立治理安全层**。

```
┌──────────────────────────────────────────────────────┐
│  前端层（React）                                        │
│  对话界面 · ECharts 渲染 · 报告面板 · Excel 上传         │
└──────────────────────────────────────────────────────┘
                    │ API / SSE（流式）
┌──────────────────────────────────────────────────────┐
│  编排层（自研；v2.1 起放弃 Dify）                        │
│  FastAPI 路由 → 编排函数（分析出图/统计/报告/KB问答）    │
│              → function-calling 循环（聊天助手，开发中）│
│                         │                              │
│                  ┌──────┴──────┐                       │
│                  │  模型路由网关  │（核心推理/视觉/轻量） │
│                  └─────────────┘                       │
└──────────────────────────────────────────────────────┘
                    │ MCP 协议（HTTP / SSE）
┌──────────────────────────────────────────────────────┐
│  MCP 工具层                                             │
│  Excel解析 · 统计分析 · 图表配置 · 报告生成              │
│  + Code Interpreter（兜底）+ 内部数据接入                │
└──────────────────────────────────────────────────────┘
                    │
┌──────────────────────────────────────────────────────┐
│  治理与安全层（所有 MCP 调用必经）                       │
│  入参 schema 校验 · 权限白名单 · 沙箱 · 可观测 · 审计     │
└──────────────────────────────────────────────────────┘
                    │
┌──────────────────────────────────────────────────────┐
│  数据存储层                                             │
│  MinIO · Redis · Milvus · PostgreSQL                   │
└──────────────────────────────────────────────────────┘
```

## 3. 模型路由设计（重点修正项）

不绑定单一模型，由**模型路由网关**按场景分发，模型通过统一接口（OpenAI 兼容 / API 适配层）接入，可随时替换。

### 3.1 模型分场景选型

| 场景 | 任务特征 | 首选模型 | 备选 |
|------|----------|----------|------|
| 核心推理 | 意图理解、分析规划、结果解读、代码生成 | **DeepSeek-V3** | Claude / GPT-4o |
| 复杂推理 | 多步规划、回归结果深度解读、疑难追问 | **DeepSeek-R1**（推理增强） | Claude |
| 多模态识图 | 图表截图理解、图像意图识别 | **Qwen2.5-VL / GLM-4V** | GPT-4o |
| 轻量任务 | query 改写、意图分类、指代消解 | **DeepSeek-V3（低温）** 或小模型 | 本地 Qwen2.5-7B |

> 说明：核心推理优先 DeepSeek（中文能力强、成本低、API 稳定）。多模态因 DeepSeek 暂无成熟视觉模型，单独配视觉模型，识图与推理解耦——视觉模型只负责"看懂图返回结构化描述"，后续推理仍回到 DeepSeek。

### 3.2 路由策略

```
请求 → 判断任务类型
  ├─ 含图像输入 → 视觉模型（识图）→ 结构化结果 → 核心推理模型
  ├─ 复杂多步（B 轨）→ DeepSeek-R1
  ├─ 标准推理 → DeepSeek-V3
  └─ 轻量分类/改写 → 低成本模型
```

- 模型配置集中管理（model registry），支持灰度切换、A/B、按租户配置。
- 统一封装重试、超时、降级（某模型不可用时切备选）。
- 记录每次调用的模型、token、耗时、成本，供治理层观测。

## 4. 中文专项设计（新增）

中文是基础设定，直接影响以下组件选型，不能套用英文默认方案。

| 环节 | 中文适配选型 | 说明 |
|------|-------------|------|
| Embedding | **bge-m3**（v2.1 已拍板，见 13.2 决策1） | 中文语义检索，稠密+稀疏双路，禁用英文默认 embedding |
| Rerank | **bge-reranker-v2-m3** | 检索结果重排，中文显著提升精度 |
| 稀疏检索分词 | **jieba** / bge-m3 内置稀疏向量 | BM25 中文必须分词，否则召回严重退化 |
| 文档解析 | PyMuPDF / unstructured + 中文 OCR（PaddleOCR） | 中文 PDF/Word 版式、扫描件 OCR |
| LLM | DeepSeek 系列 | 中文推理与口语理解强 |
| Prompt | 全中文 system prompt + 中文 few-shot | 指标口径、术语用中文示例约束 |

混合检索：向量（bge embedding）+ 稀疏（中文分词 BM25 / bge-m3 稀疏），结果融合后过 bge-reranker 重排。

## 5. 分层设计

### 5.1 前端层（React）

| 模块 | 职责 | 关键点 |
|------|------|--------|
| 对话界面 | 多轮、多模态输入 | 文本 + 图像输入，SSE 流式渲染，展示中间步骤 |
| ECharts 渲染器 | 交互图表 | 后端下发 ECharts JSON，前端渲染 |
| 报告面板 | 浏览、PDF 导出、追问入口 | 每个图表绑定底层数据 ID，可点击追问 |
| Excel 上传 | 上传、预览、确认 | 先展示数据画像供用户确认再分析 |

### 5.2 编排层（v2.1 修订：放弃 Dify，全自研）

**决定（2026-07 正式记录）**：编排不再使用 Dify，"A/B 双轨"概念退役，编排全部为自研代码。

**理由（留档备查）**：

1. **门控必须落在代码里**：业务数据进模型的安全闸门（画像脱敏、摘要白名单、小分组保护）都在自研代码中；若 Dify 承载编排，要么把明细泄给 Dify，要么在 Dify 内重造一套门控——两者都不可接受。
2. **离线局域网下 Dify 无净收益**：其知识库照样需要自部署 embedding 推理服务，环境难点是自研方案的超集，还额外背上整套平台（8~10 容器）的运维。
3. **助手主体已定自研**（第 13 章）：Dify 的低代码编排卖点用不上。

**现行编排形态**：

| 流程 | 实现 | 状态 |
|------|------|------|
| 简单流程（分析出图 / 统计 / 报告 / KB 问答） | FastAPI 路由直调编排函数（`apps/orchestrator/*`），带"错误回传重规划"兜底 | 已实现 |
| 聊天助手（自由问答，第 13 章） | DeepSeek function-calling 循环，模型自主决定查知识库 / 调统计工具 / 看画像 | 开发中 |
| 复杂多步分析 | 暂不实现；引入时再评估 LangGraph vs 自研状态机 | 后续 |

#### 5.2.1 双轨判定规则（v2.1 作废，留档）

> 以下规则随 Dify 放弃而作废，留档供未来"简单/复杂流程分流"设计参考。

1. **意图分类**（轻量模型）：纯知识问答 / 简单单图分析 → A 轨。
2. **多步信号**：请求含"先…再…然后""对比多个维度""分析后预测"等组合诉求 → B 轨。
3. **工具数量预估**：预计需 ≥3 个工具串联，或需根据中间结果决定下一步 → B 轨。
4. **追问上下文**：若追问依赖对已有分析的重新规划（非简单取数）→ B 轨。
5. 默认 → A 轨。

#### 5.2.2 多轮上下文管理

会话状态对象（**v1 内存版**，单实例 + TTL 过期，重启丢会话可接受；上 Redis 的触发条件见 13.2 决策 6）：

```
SessionState {
  session_id
  history: [turn...]            # 对话轮次（按需压缩）
  active_dataset: ref           # 当前活跃数据集引用（指向 PostgreSQL/MinIO）
  chart_registry: {chart_id → {data_ref, gen_params}}  # 图表追问用
  entity_map: {别名 → 实体ID}   # 指代消解引用表
  last_analysis: ref            # 上次分析结果
}
```

- **上下文压缩**：超出 token 阈值时，对早期轮次做摘要（LLM 滚动摘要），保留最近 N 轮原文 + 全局摘要 + 活跃数据引用。
- **指代消解**：双策略并用——
  1. 维护 `entity_map`，把"这个图""上面的结果""刚才那张表"映射到具体 `chart_id` / `dataset_ref`；
  2. 对模糊指代用轻量模型做 query 改写，把指代替换为明确实体后再进入主流程。

### 5.3 MCP 工具层

> **v2.1 形态说明（现状如实记录）**：MCP 工具当前以**进程内 `Tool.invoke`** 挂载进 API 进程，入参仍强制过 JSON Schema 校验（红线3 不变）；各工具的独立 server 进程可选启动。**MCP-over-HTTP 是可选的未来演进，不是现状**——是否演进取决于真实的独立扩缩容需求。
>
> **解读唯一出口（正式条款，v2.1 升格）**：MCP 工具本身**零 LLM**——report 的 `insight_summary` 已改为**纯拼接**（不调模型）；统计结果的 LLM 中文解读只发生在编排层（`stats_interpreter`，以及第 13 章助手通道）。**新增工具不得内嵌 LLM 调用。**

| 工具组 | 工具 | 技术栈 | 需求 |
|--------|------|--------|------|
| Excel 解析 | parse_excel / infer_schema / data_preview | pandas · openpyxl | F2 |
| 统计分析 | trend_analysis / anomaly_detect / regression | statsmodels · sklearn · Prophet | F4 |
| 图表配置 | gen_chart / chart_screenshot / multi_layout | ECharts JSON 输出 | F2 F3 |
| 报告生成 | gen_report_md / insight_summary / export_pdf | Markdown · WeasyPrint | F2 |
| Code Interpreter | 通用代码执行（兜底） | pandas + 沙箱 | F2 F4 |
| 内部数据接入 | query_db / call_api | MCP Server | F5 |

**高级统计方法落地：**

| 能力 | 实现 |
|------|------|
| 趋势分析 | STL 时序分解、移动平均、Prophet 预测 |
| 异常检测 | 3σ / IQR / Isolation Forest；时序用 STL 残差 |
| 回归分析 | statsmodels OLS/Logit，输出系数、p 值、R²、显著性，LLM 解读 |
| 相关性 | Pearson/Spearman + 热力图 |

**chart_screenshot 实现说明（补全）**：前端 ECharts 是浏览器渲染，后端截图需服务端渲染。方案：用 **Playwright 无头浏览器**加载 ECharts 配置渲染并截图，或用 echarts 的 SSR（node-canvas）服务端出图。该组件需独立部署、池化复用浏览器实例，避免每次冷启动。

### 5.4 治理与安全层

| 环节 | 职责 |
|------|------|
| 入参 schema 校验 | LLM 生成的工具入参须过 JSON Schema 校验，拦截非法/越权参数 |
| 权限 · 白名单 | 工具走白名单，数据按用户/租户权限过滤 |
| 沙箱隔离 | Code Interpreter 限制网络/文件系统/CPU/内存/超时 |
| 可观测性 | 记录每次工具调用、模型、token、耗时、成本 |
| 审计 | 内部数据接入与敏感操作留审计日志 |

### 5.5 数据存储层

> **v2.1 现状（如实记录）**：下表为目标态。当前实际：Excel/数据集用**本地 parquet + sidecar JSON**（`.data/datasets`），报告/截图落本地文件系统（`.data/reports`），KB 索引为本地 JSON（升级中），session 为**内存版**；**MinIO / Redis / PostgreSQL 均未接**。向量库按第 13 章决策上 **Milvus Lite**（pymilvus 内嵌，规模不够换 standalone 不改代码）。接入生产存储的触发条件见 13.2。

| 存储 | 目标选型 | 用途 | 当前实际 |
|------|------|------|------|
| 文件存储 | MinIO | Excel 原始文件、导出 PDF | 本地文件系统 |
| 会话缓存 | Redis | 会话状态、活跃数据引用、短期上下文 | 内存版（v1 已拍板） |
| 向量库 | Milvus | RAG 中文向量 | **Milvus Lite**（决策 2） |
| 结果存储 | PostgreSQL | 分析结果、图表元数据、报告 | 本地落盘 |

## 6. 关键流程设计

### 6.1 Excel 分析流程（F2）

```
上传 → parse_excel + infer_schema 生成"数据画像"
     → 仅把画像（列名/类型/空值率/统计摘要/样本行）喂给推理模型
     → DeepSeek 规划分析步骤 + 生成代码
     → 调用统计工具 / Code Interpreter（沙箱）执行计算
     → gen_chart 产出 ECharts JSON + insight_summary 生成中文解读
     → 报告组装并持久化（图表绑定底层 data_ref）
```

要点：**LLM 不直接处理原始数据**，只看画像；所有数值来自工具。大表超内存阈值切 **DuckDB** 分块处理。数据画像可缓存，避免同表重复计算。

### 6.2 图表追问流程（F3）

> **v2.1 状态**：本节**整体后置**——chart_registry / chart_id 当前无持久化，图表追问未实现。以下为目标设计而非现状。多轮追问能力将先由第 13 章聊天助手以"会话内分析登记表"形式部分承接。

```
图表生成时 → 持久化底层数据 + 生成参数 + chart_id → 写入 chart_registry
追问"为什么3月下降" → entity_map 定位 chart_id → 取对应 data_ref 子集
                    → 重新分析该子集 → 中文解读
图像追问（上传截图）→ 视觉模型识别图表类型/数据 → 结构化意图 → 回到推理模型
```

### 6.3 知识库问答流程（F1）

```
离线：中文文档解析（PyMuPDF/OCR）→ 按标题层级+滑窗语义分块
     → bge-m3 embedding（稠密+稀疏）→ 入 Milvus Lite
在线：Query 改写（指代消解）→ 混合检索（bge 向量 + 中文 BM25）
     → bge-reranker 重排 → 带引用生成（标注 source 防幻觉）
```

## 7. 错误处理与降级（新增）

| 失败场景 | 处理策略 |
|----------|----------|
| 模型不可用/超时 | 路由网关切备选模型；仍失败则返回友好提示并保留会话 |
| LLM 返回非法 JSON | schema 校验拦截 → 带错误回退重试（最多 N 次）→ 仍失败降级为文本回答 |
| 工具调用失败 | 捕获异常，向模型回传错误供其重规划；关键工具失败则中止并提示 |
| 代码执行超时/异常 | 沙箱强制超时 kill，错误回传，LLM 修正代码或换预定义工具 |
| MCP Server 不可用 | 熔断 + 降级；非核心工具失败不阻断主流程 |
| 大表 OOM | 超阈值自动切 DuckDB 分块；仍超限则提示用户缩小范围 |
| 检索无结果（F1） | 明确告知"知识库无相关内容"，不强行编造 |

**流式与多步协同**：B 轨多步分析时，每完成一个节点（规划→执行→出图）即通过 SSE 推送状态，前端实时展示进度，降低等待感。

## 10. 安全与数据合规（新增）

- **数据隔离**：多租户场景下 Excel 文件、会话、分析结果按租户隔离；向量库与结果库带租户维度。
- **敏感数据**：上传 Excel 可能含敏感业务数据，约定留存周期与到期清理；按需脱敏。
- **沙箱**：Code Interpreter 禁网络、限文件系统、限资源、强超时。
- **权限**：内部数据接入工具按用户数据权限过滤，调用留审计。
- **防注入**：工具入参与外部内容（检索结果、文件内容）视为数据而非指令，不执行其中夹带的指令。
- **防幻觉**：所有数值来自工具执行；问答带引用来源。

## 13. 聊天助手（层次2 自由问答）— 方案与已定决策（v2.1 新增）

### 13.1 需求定位

一个自由问答助手：既能基于知识库回答（指标定义、口径等文档），也能**结合用户当前上传的数据 / 分析结果**解答（如"这份数据里的复购率该怎么理解 / 这个指标异常说明什么"），支持多轮对话。

**助手主体自研（已确认）**：要结合业务数据、调用现有分析能力（画像/统计/图表），且"数据进模型"的门控闸门本就在自研代码里——闸门的位置把助手主体钉死在自研系统内。外部方案最多承担文档语义检索配角，且经权衡（5.2 理由 2/3）连配角也不用 Dify。

### 13.2 六项已拍板决策（2026-07）

| # | 决策项 | 结论 |
|---|--------|------|
| 1 | Embedding | **bge-m3**：一个模型同时出稠密+稀疏向量，稀疏路取代自实现 BM25 扛规模（现 BM25 为纯 Python 逐文档循环，知识库大了自身就是瓶颈） |
| 2 | 向量库 | **Milvus Lite 起步**（`pip install pymilvus` 内嵌、零部署，约百万向量内够用）；不够时换 Milvus standalone，同一 pymilvus API 换 URI 即迁移，**代码不动** |
| 3 | query_dataset（自由聚合取数工具） | **v1 不做**；上线后按真实提问中"助手算不了"的清单决定是否补做 |
| 4 | 推理 device | **必须是配置项（auto / cpu / cuda）**：本地开发用 CPU 或本地 GPU，部署到公司 GPU 服务器自动用 GPU，**切换不改代码** |
| 5 | 红线1 修订 | **助手通道例外（保守版）**：仅 `/chat` 助手通道免除白名单门控——允许把数据画像、统计工具完整结果、较多样本行直接给模型；列级 `EXCLUDE` 规则仍生效；**原有端点（analyze/stats/report）的门控行为完全不变**。详见 13.5 |
| 6 | session | **内存版 v1**（单实例、TTL 过期、重启丢会话可接受）。**上 Redis 的触发条件**：多 worker / 多实例部署，或"重启丢会话"成为真实用户痛点。接口已抽象（`SessionStore`），届时换实现不动调用方 |

### 13.3 架构与编排

```
用户消息（/chat，SSE 流式）
  → SessionState 装载（最近 N 轮原文 + active_dataset + 本会话分析登记表）
  → 组装 system + 常驻上下文（当前数据集画像，若有）
  → DeepSeek function-calling 循环（上限约 5 次工具调用）
       工具集：kb_search ｜ trend/anomaly/regression/correlation ｜ gen_chart
       每次调用：入参过 Tool.invoke schema 校验（红线3 不放松）→ 真实执行 → 结果回填
  → 最终回答（文档结论标引用来源；数据结论标"来自本数据集分析"）
  → 会话状态写回
```

要点：

- **"查知识库还是看数据"交给 function calling**，不做前置规则路由——混合问题（"这份数据里的复购率怎么理解"）必然同时要两者，规则路由必错。
- **唯一常驻注入**是当前数据集画像（几 KB，几乎每轮有用）；`kb_search` 做成工具，纯数据轮次不浪费检索。
- **防幻觉的结构化落实（红线2，任何场景都守）**：模型要新数字必须发起工具调用，数字以工具消息回填——有出处、有日志；prompt 同时约束"引用数字必须能在工具结果中找到"。
- 循环兜底：复用 analyze 链路已验证的"schema 校验失败 → 错误回传重试"模式；调用次数设上限。
- 能力边界如实告知：现有工具算不了的派生指标（复购率计算、转化漏斗），助手须明说"需要 X 口径的计算，当前工具不支持"，不含混带过（其"算不了"记录反哺决策 3）。

### 13.5 红线1 修订：助手通道例外（保守版，已拍板）

**背景**：红线1（数据与推理分离）当初按"数据可能敏感"的谨慎假设建立。实际部署为**公司局域网 + 数据不敏感**，故放松——但采用**保守版**：例外只开给助手通道，原有端点行为一字不动。

- **助手通道（/chat）**：允许把数据画像、统计工具**完整结果**（非白名单摘要）、较多样本行（如 20~50 行）直接给模型；超长结果只做 token 截断（如异常点列表截前 50 并注明），**动机是 token 经济而非安全**。列级 `EXCLUDE` 显式规则**仍生效**（配置驱动、默认全开、保住未来收紧的口子）。
- **原有端点（/analyze、/stats/*、/analyze/report）**：白名单门控、脱敏、小分组保护**完全不变**。
- **两条禁改**（写给后续开发与 AI 会话）：① 不得把助手通道"修"回严格门控；② 不得以助手例外为由放松原有端点。
- **红线2 不放松**：数值必来自工具，任何场景（含助手）都守。
- 助手发往模型的数据物料仍整体打结构化日志（可审计）。
