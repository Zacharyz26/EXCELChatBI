# 知识库部署与运维

本文覆盖知识库第三阶段（质量门禁、引用和可观测）与第四阶段（Milvus Standalone
部署、备份、切换和回滚）。原始文档目录始终是事实来源，向量索引是可重建产物。

> **当前/目标边界**：本文中的 `deploy/milvus/docker-compose.yml` 是当前已经交付且仍有效的
> Milvus 独立运维入口。全项目 Dockerfile、根目录 Compose、`knowledge-tools` MCP 容器和
> Streamable HTTP 尚未实现；它们属于 v2.4 阶段 1/2，不能使用本文的 Milvus Compose
> 代表“全项目已容器化”。详细设计见
> [`v2.4/MCP与Docker架构决策.md`](./v2.4/MCP与Docker架构决策.md)。

## v2.4 全项目容器化衔接（尚未实现）

v2.4 阶段 1 会把现有知识检索能力封装为零 LLM 的 `knowledge-tools` MCP Server，并让同一
工具契约同时通过本地 stdio 与容器内 Streamable HTTP 验证。阶段 2 再把当前 Milvus
Standalone 拓扑并入根目录 Compose 的可选 `rag` profile，同时保留本文件的独立运维入口，
避免破坏既有部署、备份和回滚流程。

容器路径需要遵守以下约束：

- `knowledge-tools` 通过 Compose 服务 DNS 访问 Milvus，不使用 `127.0.0.1` 指向其他容器；
- MCP endpoint 仅监听内部网络，不发布到公网；调试端口也只绑定宿主机 loopback；
- 知识索引、原始文档、备份和模型缓存分别挂载，模型缓存不得被当作业务备份；
- Milvus 镜像版本、鉴权、健康检查、代际 alias、持久卷及备份/恢复门槛不得因合并 Compose 而降级；
- 迁移前后必须用同一 baseline/semantic 评测集验证检索质量、引用和拒答行为等价。

在根 Compose 实际交付并通过上述验收前，以下独立 Milvus 命令仍是唯一正式容器运维方式。

## 1. 配置档位

本地开发和 CI 使用确定性替身：

```dotenv
RAG_EMBEDDER=hashing
RAG_RERANKER=lexical
RAG_STORE=local
```

真实语义检索使用：

```dotenv
RAG_EMBEDDER=bge
RAG_RERANKER=bge
RAG_STORE=milvus
MILVUS_URI=.data/milvus_lite.db
MILVUS_COLLECTION=kb_chunks
EMBEDDING_MODEL=BAAI/bge-m3
RERANK_MODEL=BAAI/bge-reranker-v2-m3
RAG_MIN_RELEVANCE=0.02
```

`RAG_EMBEDDER=bge` 与 `RAG_STORE=milvus` 必须成对启用。非法枚举、空模型名、越界阈值
会在 API 启动时直接报错。连接本机 Milvus 时，客户端自动维护 `NO_PROXY/no_proxy` 的
loopback 豁免。

## 2. 文档生命周期与一致性

每个来源按 source 稳定映射为 `document_id`，并记录原文 SHA-256 `content_hash`、版本、
更新时间和片段数：

- source 不变、哈希不变：增量同步跳过；
- source 不变、哈希变化：替换该文档全部片段，版本加一；
- source 新增：创建版本 1；
- 全量重建：输入目录中已消失的 source 会从新索引删除；
- 前端上下文面板可查看来源、版本、片段数，并按文档删除。

目录摄入使用相对 `KB_DOCS_DIR` 的路径作为 source，避免不同子目录同名文件冲突。单次
文件数和单文档字符数由 `KB_MAX_FILES`、`KB_MAX_DOCUMENT_CHARS` 限制。

```bash
# 默认读取 KB_DOCS_DIR，仅处理新增/变化文档
uv run python scripts/kb_rebuild.py --mode incremental

# 全量构建新索引，成功后才发布
uv run python scripts/kb_rebuild.py --mode full

# CLI 可读取指定可信目录；HTTP 入口仍只允许 KB_DOCS_DIR 内路径
uv run python scripts/kb_rebuild.py --mode full --path /srv/chatbi/kb-docs
```

HTTP 对应入口为 `POST /kb/ingest`、`POST /kb/rebuild`、
`DELETE /kb/documents/{document_id}` 和 `GET /kb/overview`。

全量/增量更新都会先完成 embedding，再改动索引。Local 后端使用临时 JSON +
`os.replace`；Milvus 先创建并加载新的物理集合，然后发布活动集合：Lite 使用数据库旁的
原子 JSON 指针，Standalone 使用 `${MILVUS_COLLECTION}_active` alias。构建、写入或加载
失败时，原活动索引继续服务；并发读只会看到完整的旧快照或新快照。

## 3. 质量门禁、引用与可观测

CI 使用固定 hashing/local 档执行基线，避免下载模型造成结果漂移；生产发布前另跑 semantic
档。两档都在临时隔离索引上重建评测语料，不会污染已配置的业务集合；semantic 档验证真实
BGE 模型与 Milvus schema/检索链路，Standalone 连接另由 readiness 和负载冒烟验证。
`--enforce` 未达阈值时退出码为 1，`--json-output` 生成可归档报告：

```bash
# CI/离线确定性基线
RAG_EMBEDDER=hashing RAG_RERANKER=lexical RAG_STORE=local \
  RAG_MIN_RELEVANCE=0 \
  uv run python scripts/kb_eval.py --enforce --profile baseline \
  --json-output .data/kb-eval.json

# 真实 BGE 质量验收（临时 Milvus Lite 索引）
uv run python scripts/kb_eval.py --enforce --profile semantic \
  --json-output .data/kb-eval-semantic.json
```

报告包括 lexical/semantic hit@1、hit@3、MRR、负例拒答率、引用来源完整率、分数分布和
查询延迟。JSON 不包含原始问题文本，只记录用例编号和来源。CI 会将其作为
`kb-evaluation` Artifact 保存 30 天。

检索日志事件为 `rag.retrieve`，包含总耗时、embedding/dense/sparse/rerank 分段耗时、各路
候选数、top score 和拒答原因；同步日志事件为 `rag.sync`，包含新增/更新/跳过/删除计数和
耗时。日志不记录问题原文或文档正文。Agent 的 `kb_search` 结果会持久化为 `citations`
Artifact，前端显示来源、小节与原文片段，刷新后仍可回放。

就绪探针真实读取当前存储状态，并且错误响应不泄露 URI、token 或异常正文：

```bash
curl --fail http://127.0.0.1:8000/health/ready
uv run python scripts/kb_admin.py status
```

## 4. Milvus Standalone 部署

仓库内 `deploy/milvus/docker-compose.yml` 固定 Milvus `v2.6.20`，默认仅监听
`127.0.0.1`，开启鉴权、持久化目录、依赖健康检查和自动重启。目标机需先安装 Docker
Engine 与 Compose 插件，并保证 etcd 所在磁盘具有稳定低延迟。

```bash
cp deploy/milvus/.env.example deploy/milvus/.env
# 编辑 .env：至少替换两个 MinIO 凭据；不要提交此文件

docker compose --env-file deploy/milvus/.env \
  -f deploy/milvus/docker-compose.yml config --quiet
docker compose --env-file deploy/milvus/.env \
  -f deploy/milvus/docker-compose.yml up -d
docker compose --env-file deploy/milvus/.env \
  -f deploy/milvus/docker-compose.yml ps
curl --fail http://127.0.0.1:9091/healthz
```

首次启动后立即创建业务账号并轮换默认 root 密码。密码只通过临时环境变量传入，脚本输出
不会包含密码：

```bash
export MILVUS_BOOTSTRAP_TOKEN='root:Milvus'
export MILVUS_APP_PASSWORD='替换为强随机密码'
export MILVUS_NEW_ROOT_PASSWORD='替换为另一强随机密码'
uv run python scripts/milvus_bootstrap.py
unset MILVUS_BOOTSTRAP_TOKEN MILVUS_APP_PASSWORD MILVUS_NEW_ROOT_PASSWORD
```

业务账号获得 `CollectionAdmin`。由于应用会动态创建带代际后缀的物理集合并切换 alias，
授权范围默认为 `*`；因此该 Compose 实例应专用于 ChatBI，不与其他业务共用。然后复制
`deploy/.env.production.example` 到部署系统的密钥配置，设置
`MILVUS_TOKEN=chatbi:<MILVUS_APP_PASSWORD>`，不要把真实值写入仓库。

模型权重建议预先侧载到 `EMBEDDING_MODEL`、`RERANK_MODEL` 指定目录，并设置
`HF_HUB_OFFLINE=1`。启动 API 后依次执行：

```bash
uv run python scripts/kb_rebuild.py --mode full
curl --fail http://127.0.0.1:8000/health/ready
uv run python scripts/kb_eval.py --enforce --profile semantic
uv run python scripts/kb_load_test.py --requests 50 --concurrency 2 \
  --max-p95-ms 2000 --json-output .data/kb-load.json
```

GPU 模型首次推理包含预热成本；先执行一次查询，再把负载结果作为该机器的基线。并发数应
逐步增加，观察 GPU 显存和 Milvus 延迟，不要直接照搬示例作为容量结论。

## 5. 发布、状态、清理与回滚

每次发布按以下顺序进行：

1. 备份原始文档和当前 Milvus 数据；
2. `kb_rebuild.py --mode full` 构建并原子切换新代；
3. 检查 `/health/ready`、`kb_admin.py status`；
4. 执行 semantic 质量门禁和只读并发冒烟；
5. 浏览器实测知识库提问，确认答案正文与引用卡同时出现且刷新后保留；
6. 稳定观察后清理旧代，但至少保留当前代和上一代。

```bash
# 查看 active / previous / generations，不输出连接密钥
uv run python scripts/kb_admin.py status

# 验收失败时原子切回上一代；再次执行可切回
uv run python scripts/kb_admin.py rollback --yes

# 清理非活动历史代；active 和 previous 不会被删除
uv run python scripts/kb_admin.py cleanup --retain 2 --yes
```

回滚只切索引，不还原原始文档目录。回滚后应再次跑 readiness、semantic 评测和浏览器引用
检查。Local 后端没有代际 alias，需通过离线备份恢复。

## 6. 备份与恢复

Local/Milvus Lite 必须先停止 API/Lite 进程，显式确认后才允许操作。恢复会先校验 manifest
和所有 SHA-256，再原子覆盖，校验失败不会修改当前索引：

```bash
uv run python scripts/kb_admin.py backup --service-stopped
uv run python scripts/kb_admin.py restore \
  --input /srv/chatbi/backups/kb/<timestamp> --service-stopped --yes
```

Standalone 不应复制运行中的 volume。使用官方 `milvus-backup` 对集合做一致性备份，并定期
做独立环境恢复演练；同时备份 `KB_DOCS_DIR`、Compose 配置、模型版本与评测报告。任何备份
都不得包含已提交到仓库的真实 token。

## 7. 停机与故障边界

Milvus Lite 适合单进程和中小规模本地部署，并对数据库文件持独占锁。同一 `.db` 不要同时
由两个后端进程占用；测试使用临时 URI。不要复制 Lite 数据库文件作为 Standalone 迁移
手段，迁移应从原始文档执行一次全量重建。

停止 Standalone：

```bash
docker compose --env-file deploy/milvus/.env \
  -f deploy/milvus/docker-compose.yml stop
```

`docker compose down` 不会主动删除绑定目录，但运维时仍禁止使用 `down -v` 或手工删除
`MILVUS_DATA_DIR`。磁盘告警、etcd 不健康、MinIO 不健康或 `/health/ready` 返回 503 时停止
发布，先保留现场和备份，再恢复上一套服务/索引。
